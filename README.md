# Тестовое задание для Fort Telecom

## ✒️Описание приложения
Приложение логически разделено на два основных модуля: сервер и клиент.  

Сервер представляет собой демона, который по средствам сокетов обрабатывает запросы клиента, 
а также собирает статистику сетевого интерфейса с определенной периодичностью. Имя сетевого интерфейса и периодичность указываются в файле uci-конфигурации.  
**Для корректной работы необходимо наличие файла конфигурации в формате uci в /etc/config/**

Клиент может отправлять два типа запросов: get и set.  
Результатом get-запроса является сетевая статистика, такая как **имя сетевого интерфейса, период обновления, rx_bytex, tx_bytes, ip, mac**.   

C помощью set-запроса можно переопределить сетевой интерфейс и период обновления сетевой статистики. Отправляя запрос этого формата, произойдет изменение файла uci-конфигурации.  
Форматы set-запроса: **set ifname=eth0 time=10**, **set ifname=eth0**, **set time=10**  

## 🏗️Архитектура проекта   

Проект следует **модульной структуре**, где каждый компонент инкапсулирует свою ответственность:

|    Модуль    | 📄 Файлы | 🎯 Задача |
|---------------|----------|-----------|
| **👻 daemon** | `daemon.c` | Создание демона, цикла событий. Запуск сервера, таймера. Обработка событий |
| **💻 server** | `server.{c,h}` | Инициализация сервера, обработка get и set запросов |
| **📱 client** | `client.{c,h}` | Подключение к серверу, отправка get и set запросов |
| **📊  netstats** | `netstats.{c,h}` | Получение статистики сетевого интерфейса |
| **🔧 uci_loader**   | `uci_loader.{c, h}` | Загрузка uci конфигурации и ее сохранение |
| **🛠️ config** | `config.h` | Определение констант и структуры самого конфига | 

## 📐Логика работы
**Сервер:**
  1. Создание демона
  2. Загрузка конфигурации 
  3. Чтение сетевой статистики на основе загруженной конфигурации
  4. Создание серверного сокета
  5. Создание таймера с интервалом из конфигурации
  6. Создание цикла событий и передача в него дескрипторов серверного сокета и таймера для обработки
  7. Ожидание срабатывания одного из событий (сообщение от клиента, срабатывание таймера)

  Событие: сообщение от клинта    
  1. get-запрос: вывод сетевой статистики  
  2. set-запрос: изменение сетевого интерфейса и указание нового периода для таймера  

  Cобытие: Срабатывание таймера      
  1. чтение сетевой статистики на основе конфигурации  

**Клиент:**
  1. Подключение к серверу
  2. Выполнение get, set запроса
  3. Вывод результата в консоль

> 💡 `*`Клиент и сервер пишут свои собственныe логи в отдельные файлы: *server_log.txt* и *client_log.txt*, находящиеся в */var/log*

## 🔨Компиляция 
### 📋Требования
- Компилятор gcc (или совместимый)  
- Библиотеки libubox и uci  
- make  
- ОС: Linux

> ❗ Поскольку libubox и uci являются компонентами OpenWrt и в основной базе данных пакетного менеджера Archlinux их нет, пришлось скачать и скомпилировать их вручную, чтобы они попали в /usr/local/lib
```` bash
 git clone https://git.openwrt.org/project/libubox.git 
 cd libubox && mkdir build && cd build  
 cmake ..  
 make -j$(nproc)  
 sudo make install
 sudo ldconfig  
````
Компиляци проекта
```` bash
#компиляция сервера
make server

#компиляция клиента
make client

#компиляции клиента и сервера
make all

#удаление исполняемых файлов
make clean
````
> 💡 `*`В результате получится два исполняемых файла: server_app и client_app. Лучше запускать их под sudo.


## 🐳 Docker
Можно не устанавливать библиотеки самостоятельно и воспользоваться Dockerfile, который развернет ubuntu и запустит на ней сервер

Находясь в корне проекта выполнить 
```` bash
#Создание образа
docker build -t ftelecom .

#Запуск контейнера
docker run -d --name server -p 8080:8080 ftelecom

#Запуск клиента в контейнере
docker exec -it server sudo /opt/ftelecom/client_app get
docker exec -it server sudo /opt/ftelecom/client_app set ifname=tun0 time=50
````
